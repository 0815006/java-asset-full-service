# 数据库操作与表结构指南

本指南详细说明了资产管理系统后端的数据库表结构、核心字段及其业务含义，旨在帮助开发和运维人员理解数据模型。

## 1. 核心设计理念
- **逻辑删除**: 所有核心业务表均采用 `is_deleted` 字段进行逻辑删除，确保数据可追溯。
- **乐观锁**: (未来规划) 可在关键资源表（如 `asset_file`）中增加 `version` 字段，用于实现乐观锁，防止并发修改冲突。
- **物化路径**: `asset_file` 表使用 `tree_path` 字段存储层级关系的完整路径，避免了递归查询，极大提升了树形结构的查询性能。

## 2. 核心业务表说明

### 2.1 `user` - 用户表
存储系统的登录用户信息。
- **`id`**: 主键。
- **`username`**: 登录账号，唯一。
- **`password_hash`**: 存储加密后的密码。
- **`real_name`**: 用户真实姓名，用于前端水印等防泄露功能。
- **`emp_no`**: 用户工号，同样用于水印。
- **`role_type`**: 全局角色，决定了用户的基本权限级别 (1=管理员, 2=测试经理, 3=普通用户)。

### 2.2 `product` - 产品表
定义了资产的顶层隔离域“产品”。
- **`id`**: 主键。
- **`product_name`**: 产品的名称，如“核心交易平台”。
- **`team_name`, `domain_name`**: 产品的组织归属，用于在首页进行分组展示。
- **`owner_id`**: 指向 `user.id`，表示该产品的负责人。
- **`asset_count`**: 冗余字段，用于在前端快速展示该产品的资产总数，避免实时计算。

### 2.3 `asset_file` - 核心资产树表
这是系统最核心的表，以无限级树形结构存储所有文件和文件夹。
- **`id`**: 主键，也是文件/文件夹的唯一标识。
- **`product_id`**: **核心隔离字段**。
  - `> 0`: 表示该条目属于某个具体的产品，值为 `product.id`。
  - `= 0`: 表示该条目属于公共专区（如“测试技术及工艺专区”或“测试管理专区”）。
- **`parent_id`**: 父节点 ID。如果为 `0`，表示它是其所属产品或专区的根节点。
- **`tree_path`**: **性能关键字段**。存储从根节点到当前节点的完整 ID 路径，格式如 `/0/1001/1002/`。
  - **查询某目录下所有子孙节点**: `WHERE tree_path LIKE '/0/1001/1002/%'`。
- **`node_type`**: `1` 表示文件夹，`2` 表示文件。
- **`file_name`**: 文件或文件夹的名称。
- **`is_latest`**: **版本控制字段**。`1` 表示这是最新版本，`0` 表示是历史版本。查询时通常需要带上 `WHERE is_latest = 1`。
- **`local_path`**: 文件在服务器上的物理存储绝对路径。
- **`solr_id`**: 对应 Solr 索引中的文档 ID，通常与 `id` 一致。

### 2.4 `user_favorite_product` - 用户收藏表
用户和产品之间的多对多关系表，用于记录用户在首页收藏的产品。
- **`user_id`**: 用户 ID。
- **`product_id`**: 产品 ID。

### 2.5 `edit_lock` - 并发编辑锁表
用于实现在线编辑（如 Markdown）时的并发控制，防止多人同时修改导致内容覆盖。
- **`asset_file_id`**: 被锁定的文件 ID，有唯一约束，确保一个文件同时只能被一个锁锁定。
- **`lock_ticket`**: 一个 UUID，作为释放锁的凭证，防止误释放。
- **`locked_by`**: 持有锁的用户 ID。
- **`expires_at`**: 锁的过期时间。前端需要通过心跳请求来不断“续租”这个锁。

## 3. 关键业务流程的数据操作

- **查询产品首页的目录树**:
  `SELECT * FROM asset_file WHERE product_id = ? AND parent_id = 0 AND is_latest = 1;`
- **懒加载查询子目录**:
  `SELECT * FROM asset_file WHERE parent_id = ? AND is_latest = 1;`
- **更新文件版本**:
  1. `UPDATE asset_file SET is_latest = 0 WHERE id = <旧文件ID>;`
  2. `INSERT INTO asset_file (...) VALUES (..., version_no = <旧版本+1>, is_latest = 1, ...);`
- **移动文件/文件夹**:
  1. 更新当前节点的 `parent_id` 和 `tree_path`。
  2. 查询出所有子孙节点 (`WHERE tree_path LIKE '<旧tree_path>%'`)，并循环更新它们的 `tree_path` 前缀。
